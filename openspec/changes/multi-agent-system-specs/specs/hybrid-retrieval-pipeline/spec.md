## ADDED Requirements

### Requirement: 向量檢索執行路徑
系統 SHALL 實現純向量檢索路徑：接收查詢向量（512 維或 1024 維），在 LanceDB 中執行第一階段粗檢索（Top-K_1 = 500）及第二階段 MaxSim 精檢索（Top-K_2 = 20），返回排序的候選投影片列表。

#### Scenario: 向量查詢執行
- **WHEN** 接收查詢向量（例如「金融科技」文本編碼後的向量）
- **THEN** 系統依次執行 IVF 篩選（產生 500 候選）→ MaxSim 精算（產生 20 排序結果），返回 RetrievalResult

#### Scenario: 整數量化向量匹配
- **WHEN** 語料庫向量已量化為 8-bit，查詢向量為 float32
- **THEN** 系統自動量化查詢向量至 8-bit，執行 8-bit 距離計算，返回結果未量化（恢復精度）

#### Scenario: 向量檢索效能 SLA
- **WHEN** 在百萬級語料上執行向量檢索
- **THEN** 第一階段延遲 < 50ms，第二階段延遲 < 100ms，總延遲 < 200ms（端到端）

### Requirement: 全文檢索（FTS）執行路徑
系統 SHALL 實現全文檢索路徑：從查詢文本提取關鍵詞，在 LanceDB 的 FTS 索引上執行模糊匹配與精確匹配，返回相關投影片的元資料層面的匹配結果（例如投影片標題、備註中包含關鍵詞的投影片）。

#### Scenario: 關鍵詞提取
- **WHEN** 查詢「深度學習在醫療診斷的應用」
- **THEN** 系統提取核心關鍵詞：「深度學習」、「醫療」、「診斷」、「應用」

#### Scenario: FTS 索引匹配
- **WHEN** 對關鍵詞執行 FTS 查詢
- **THEN** 系統檢索投影片元資料，找出標題或備註中包含這些關鍵詞的投影片，例如「xx.pptx, slide #5: Deep Learning for Diagnosis」

#### Scenario: 模糊匹配容錯
- **WHEN** 查詢包含拼寫錯誤（如「深的學習」）
- **THEN** FTS 索引執行模糊匹配（edit distance < 2），仍能找到「深度學習」相關投影片

### Requirement: 檢索結果融合策略
系統 SHALL 實現向量 + FTS 的融合，使用加權融合公式：
final_score = α × normalized_vector_score + β × normalized_fts_score
其中 α + β = 1，預設 α = 0.7，β = 0.3（可配置），向量得分與 FTS 得分各自歸一化至 [0, 1] 區間。

#### Scenario: 預設權重融合
- **WHEN** 查詢同時包含文本與隱含的語義
- **THEN** 系統執行向量檢索（得分 v_score = 0.85）與 FTS 檢索（得分 f_score = 0.60），融合 final = 0.7 × 0.85 + 0.3 × 0.60 = 0.775

#### Scenario: 動態權重調整
- **WHEN** 使用者指定「精確匹配優先」模式
- **THEN** 系統調整權重 α = 0.5，β = 0.5，提高 FTS 權重，結果排序更傾向字面匹配

#### Scenario: 單路徑退化
- **WHEN** FTS 返回結果為空（無匹配關鍵詞）
- **THEN** 系統自動降級為純向量檢索，β = 0，α = 1，直接使用 v_score

### Requirement: 多構面查詢支持
系統 SHALL 支援多構面（faceted）查詢，例如用戶可指定「講者」、「日期」、「主題」等元資料維度進行篩選，系統在檢索過程中組合向量 + 元資料過濾器，提高結果精準度。

#### Scenario: 単一構面篩選
- **WHEN** 使用者查詢「機器學習」同時篩選 speaker = "Alice"
- **THEN** 系統先執行向量檢索（找出相關投影片），再過濾出 speaker 欄位包含「Alice」的投影片

#### Scenario: 多重構面組合
- **WHEN** 使用者篩選 date >= "2024-01-01" AND topic IN ["AI", "ML"]
- **THEN** 系統基於多個元資料條件進行過濾，結合向量相似度排序，返回融合結果

#### Scenario: 構面統計
- **WHEN** 完整檢索後
- **THEN** 系統返回各構面的統計信息（例如「相關投影片中，Alice 發表 15 份，Bob 發表 8 份」），支援交互式鑽探

### Requirement: 查詢路由與智能選擇
系統 SHALL 實現查詢路由邏輯，根據查詢特徵（是否包含文本、是否包含圖像、文本長度）自動選擇最優檢索路徑：
- 純文本（< 100 字）→ 向量 + FTS 融合（α=0.7, β=0.3）
- 純文本（> 100 字）→ 向量優先（α=0.9, β=0.1）
- 純圖像 → 向量檢索（α=1.0，無 FTS）
- 混合 → 融合融合（α=0.6, β=0.4）

#### Scenario: 短文本查詢路由
- **WHEN** 查詢「AI 應用」（5 詞）
- **THEN** 系統判定為短文本，選擇 α=0.7, β=0.3 的融合路徑

#### Scenario: 長文本查詢路由
- **WHEN** 查詢為完整段落，描述投影片內容需求（200+ 詞）
- **THEN** 系統判定為長文本，選擇 α=0.9, β=0.1 路徑，減少 FTS 權重（因長文本很難完全匹配）

#### Scenario: 圖像查詢路由
- **WHEN** 使用者上傳參考圖像，無文本描述
- **THEN** 系統選擇純向量檢索路徑，α=1.0

### Requirement: 檢索結果去重與多樣性排序
系統 SHALL 支援檢索結果的去重（相同或高度相似的投影片只保留一份）及多樣性重排（Diversity Ranking），確保返回的結果在內容上有一定的差異性，避免結果過度重複。

#### Scenario: 近似重複檢測
- **WHEN** 檢索結果包含兩個向量余弦相似度 > 0.95 的投影片
- **THEN** 系統識別為重複，僅保留得分更高的那份，移除低分重複

#### Scenario: 多樣性重排
- **WHEN** 檢索返回 20 個結果，使用者開啟「多樣性優先」選項
- **THEN** 系統進行多樣性重排（例如 MMR - Maximal Marginal Relevance），平衡相關性與多樣性，確保結果覆蓋多個子主題

#### Scenario: 主題叢集參數化
- **WHEN** 結果包含多個子主題（如「機器學習」、「深度學習」、「自然語言處理」）
- **THEN** 系統按主題簇進行分層返回，允許使用者按需探索各簇

### Requirement: 檢索上下文與會話融合
系統 SHALL 支援檢索會話上下文，記住用戶的過往查詢與反饋，用於改善後續查詢的結果排序。例如用戶可標記某結果為「相關」或「不相關」，系統利用此反饋調整檢索策略。

#### Scenario: 會話歷史追蹤
- **WHEN** 使用者連續提交 3 個檢索查詢
- **THEN** 系統在會話中保存查詢歷史與點擊反饋，上下文 ID 用於連結相關查詢

#### Scenario: 動態排序權重調整
- **WHEN** 用戶對第一輪結果標記「相關」與「不相關」
- **THEN** 系統學習用戶偏好，在第二輪查詢中調整檢索權重，例如降低不相關投影片的演講者權重

### Requirement: 效能與可靠性
系統 SHALL 達成：混合檢索端到端延遲 < 250ms，單一檢索路徑成功率 > 99.9%，降級轉換延遲 < 50ms（例如 FTS 失敗時快速切換至純向量検索）。

#### Scenario: 混合檢索延遲 SLA
- **WHEN** 執行向量 + FTS 融合檢索
- **THEN** 在 < 250ms 內返回結果，向量檢索 < 150ms，FTS < 100ms

#### Scenario: 失敗自動降級
- **WHEN** FTS 索引查詢超時（或不可用）
- **THEN** 系統自動降級至純向量檢索，< 50ms 內完成轉換，用戶無感知

### Requirement: 輸出介面標準化
系統 SHALL 返回統一的 `HybridRetrievalResult`，包含混合檢索的排序候選列表、各檢索路徑的得分（vector_score、fts_score、final_score）、使用的融合權重。支援結果導出（JSON、CSV）。

#### Scenario: 混合結果結構
- **WHEN** 混合檢索完成
- **THEN** 返回 result.ranking 列表，每項含 slide_id、vector_score、fts_score、final_score、fusion_weights (α, β)

#### Scenario: 檢索透視
- **WHEN** 使用者查看檢索詳情
- **THEN** 系統展示該查詢使用的檢索路徑（「向量 70% + FTS 30%」）與各檢索路徑的中間結果
