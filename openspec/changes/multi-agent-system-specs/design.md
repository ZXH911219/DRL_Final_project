## Context

傳統簡報檢索系統依賴 OCR 文字提取與稀疏向量比對，存在以下核心問題：
- **語義理解不足**：無法捕捉投影片中複雜的視覺元素（圖表、排版、視覺層級）與語義關係
- **幻覺率高**：推理模型生成的結果常缺乏真實視覺證據支撐，企業無法信任
- **檢索精度低**：Recall@10 通常 < 70%，NDCG < 0.65，難以滿足企業需求
- **不可解釋性**：黑盒式的向量相似度排序缺乏人類可理解的推理過程

產業最新突破（ColPali, ImageBind, MM-R5, Argos）提供了解決方案，但這些技術的整合不平凡：
- ColPali 產生的多向量（1024 維向量 × 128 數值）儲存與檢索成本龐大
- 多模態特徵空間對齊的複雜度高，需細心管理空間映射
- 推理模型的延遲（1-3 秒/候選）難以在高吞吐場景中接受
- 視覺驗證層的幻覺偵測涉及複雜的空間定位與語義一致性檢查

本設計將這四項技術組織為專業化的多代理架構，透過分層責任、異步通訊與容錯機制，實現企業級的精準、快速、可信簡報檢索。

## Goals / Non-Goals

**Goals:**
- 設計一個四層多代理架構，每個代理專注單一責任，通過標準化接口協作
- 實現毫秒級（< 200ms）端到端檢索延遲，支援百萬級投影片語料
- 提供完整的思維鏈推理過程，推理透明度 > 90%
- 通過視覺驗證層消除視覺幻覺，確保 100% 的結果準確性
- 建立完整的審計日誌与可視化證據地圖，支援決策審計追蹤
- 支援異質模態查詢（文字、圖像、混合），利用統一的多模態向量空間
- 提供可靠的容錯與降級策略，系統可用性 > 99.5%

**Non-Goals:**
- 不實現自動標籤生成或投影片內容摘要（留給上層應用）
- 不支援即時投影片編輯同步（本系統採用批量離線索引）
- 不提供機器學習模型微調的自動化工具（模型權重應由 MLOps 層管理）
- 不實現跨語言投影片檢索（初版本針對中英文優化）
- 不支援 PPT 之外的簡報格式（如 Keynote 格式），未來可擴展

## Decisions

### 1. 四層代理分層架構 vs. 單一端到端模型

**決定**：採用四層多代理架構（Vision → Retrieval → Reasoning → Verification）

**理由**：
- **模塊化與可維護性**：每層獨立演化，測試與優化相對隔離
- **性能優化機會**：檢索層可透過索引加速，推理層可透過模型蒸餾優化，驗證層可透過多進程並行化
- **容錯粒度細**：某一代理失敗可局部降級，無需整體重新計算
- **技術選型自由度高**：未來可輕鬆替換或升級某一層的模型，無需重構其他層

**備選方案檢視**：
- 端到端單一模型：可減少通訊開銷，但難以達到各層的性能目標，且幻覺率高於分層驗證的方案

### 2. LanceDB 雙層檢索 vs. FAISS/Pinecone

**決定**：採用 LanceDB 作為檢索核心，實現雙層檢索（向量篩選 + MaxSim 精比對）

**理由**：
- **磁碟型存儲**：LanceDB 的磁碟存儲支援百萬級向量，成本遠低於全內存方案（FAISS）
- **多向量支援**：LanceDB 原生支援多向量檢索與 Late Interaction，相比 Pinecone 的向量維度限制更靈活
- **開源生態**：LanceDB 與 Apache Arrow 深度整合，支援高效的 Parquet 序列化與 GPU 加速
- **混合檢索**：LanceDB 支援向量 + 全文檢索（FTS）的靈活融合

**備選方案檢視**：
- Pinecone：託管服務降低運維成本，但成本高（按查詢計費），難以支援企業級百萬級規模
- FAISS：開源免費，但需自行搭建叢集與負載均衡，且全內存架構不適合超大規模

### 3. MM-R5 多模態推理 vs. 基於提示的 GPT-4

**決定**：採用開源的 MM-R5 模型（~7B 參數），在本地或專有雲部署，而非調用 GPT-4 API

**理由**：
- **延遲控制**：MM-R5 本地推理 1-3 秒/候選，延遲可控且可預測。GPT-4 API 延遲受限於網路與 API 配額，不確定性高
- **成本節制**：按候選計費（100 多詞推理），百萬級語料的推理成本難以承受。本地部署成本固定
- **隱私與安全**：投影片內容不上傳至第三方服務，符合企業合規要求
- **可重現性**：開源模型可精確復現結果，便利審計與調試

**備選方案檢視**：
- GPT-4 / Claude API：推理品質更高，但成本與延遲均無法控制，不適合企業生產環境

### 4. Argos 視覺驗證 vs. 簡單置信度閾值

**決定**：實現完整的 Argos 視覺驗證框架，包括推論-視覺映射檢查、幻覺風險評分與審計日誌

**理由**：
- **零幻覺承諾**：簡單置信度閾值無法捕捉語義矛盾（例如：推理說「增長趨勢」但圖表顯示「下降」）。Argos 的空間定位驗證可偵測此類矛盾
- **不可否認的審計痕跡**：每份結果都有詳細的驗證過程與決策理由，符合企業審計要求
- **動態風險調整**：幻覺風險分數 ∈ [0, 1]，可動態調整結果權重，而非二值通過/不通過

**備選方案檢視**：
- 簡單置信度閾值：實現簡單，但無法處理複雜的幻覺場景，最終錯誤率仍>10%

### 5. ImageBind 統一空間 vs. 模態特定編碼器

**決定**：利用 ImageBind 將所有模態（文字、圖像、音頻等）映射至統一的共享向量空間

**理由**：
- **跨模態檢索**：使用者可用文字查詢投影片，或上傳參考圖像自動查找相似內容
- **特徵空間一致性**：統一空間確保不同模態間的語義距離度量一致，易於融合
- **未來擴展性**：若未來需支援音頻查詢（例如演講錄音片段的投影片檢索），可無縫擴展

**備選方案檢視**：
- 模態特定編碼器：需維護多個獨立的向量空間與跨空間映射函數，複雜度高且融合策略需人工調整

### 6. 異步消息隊列 vs. 同步 RPC 調用

**決定**：採用異步消息隊列（如 RabbitMQ / Kafka）作為代理間通訊主幹，同時提供同步 gRPC 端點供外部調用

**理由**：
- **解耦與伸縮性**：代理間解耦，各層可獨立伸縮。若推理層出現瓶頸，無需修改檢索層
- **容錯與重試**：消息隊列提供天然的重試機制與死信隊列，便於容錯處理
- **吞吐量最大化**：非阻塞通訊，檢索層可在檢索結果返回後，後台異步啟動推理與驗證
- **審計追蹤**：消息日誌可作為完整的呼叫鏈追蹤與審計記錄

**備選方案檢視**：
- 純同步 RPC：延遲累加（檢索 50ms + 推理 1500ms + 驗證 500ms → 2050ms），難以滿足 < 200ms 的檢索 SLA
- 純異步隊列：對於簡單的檢索查詢，異步開銷浪費，同步端點更直接

## Risks / Trade-offs

### [Risk 1] MultiVector 儲存成本膨脹
- **描述**：每頁投影片產生 1024 × 128 = 131,072 個浮點數，百萬級語料總量達 TB 級，儲存與 I/O 成本龐大
- **緩解策略**：
  - 向量量化（8-bit 或 16-bit）可減少 75-87% 儲存成本，精度損失在 < 2% 內
  - 內存映射文件與磁碟索引優化，避免全量加載至內存
  - 分佈式索引與分片策略，將索引分割至多臺機器

### [Risk 2] MM-R5 推理延遲（1-3 秒/候選）
- **描述**：若初步檢索返回 20 個候選，推理層需 20-60 秒方能完成，遠超預期的端到端 SLA
- **緩解策略**：
  - 後台異步推理：檢索結果優先返回，推理與驗證在後台進行
  - 模型蒸餾：將 MM-R5（7B）蒸餾為更小的模型（3B），推理延遲可降至 0.5-1 秒
  - 候選預篩選：對初步檢索結果進行快速的語義相似度預篩選，只對前 5 個候選進行深度推理

### [Risk 3] 視覺驗證的假陽性率（誤判為幻覺）
- **描述**：OCR 識別精度不完美，或投影片文字與圖表排版複雜，驗證模型可能誤判為幻覺
- **緩解策略**：
  - 多層次證據甄別：若一項聲明無法通過 OCR 驗證，再次嘗試視覺識別（圖像 CLIP 編碼）
  - 置信度下限設定：幻覺風險 < 0.15 時，不進行降權，允許一定的不確定性
  - 人工反饋迴圈：使用者標記「誤判」的案例，用於離線微調驗證模型

### [Risk 4] 多模態特徵空間對齊的語義漂移
- **描述**：ImageBind 的統一向量空間可能對不同模態的語義邊界處理不一致，導致跨模態查詢精度下降
- **緩解策略**：
  - 對齊品質監控：定期計算跨模態相似度的 Consistency Score，若 < 0.85 啟動告警
  - 空間微調：針對簡報領域特異性，收集小規模標註數據（~10K，投影片 + 文字描述對），微調 ImageBind
  - 模態優先級設定：若跨模態精度無法達到，可按「文字優先 > 圖像 > 混合」的優先級遞減

### [Risk 5] 代理通訊的網路分割 (Network Partition)
- **描述**：若消息隊列或代理間通訊出現網路故障，可能導致檢索結果返回但推理/驗證無法完成
- **緩解策略**：
  - 異步化的結構冪等性：消息可安全重試，代理端點均實現幂等設計
  - 死信隊列與監控：未能成功完成的推理/驗證任務進入死信隊列，觸發告警與人工介入
  - 快速降級：檢索結果可在無推理/驗證的情況下返回，但標記可信度為「部分」

### [Trade-off] 即時性 vs. 精確性
- **折衝**：全流程（檢索 + 推理 + 驗證）端到端延遲 ~2 秒，對實時交互式查詢而言較長
- **決策**：優先選擇精確性與可信度，接受 2 秒的延遲是企業決策應用（例如編輯選擇）的可接受成本
- **改善方案**：若未來需支援 < 500ms 的快速預覽模式，可提供「無驗證」或「無推理」的降級版本

## Migration Plan

### Phase 1: Infrastructure Readiness（第 1-2 週）
1. 部署 LanceDB 集群、消息隊列（RabbitMQ 或 Kafka）、監控儀表板（Prometheus + Grafana）
2. 配置 GPU 計算資源（推理與驗證層各 1-2 張 GPU），訂閱必要的模型權重（ColPali, MM-R5, ImageBind）
3. 建立開發、測試、生產環境的隔離與漸進式灰度發佈機制

### Phase 2: Agent Implementation（第 3-8 週，並行執行）
- **Vision-Ingestion-Agent**（第 3-4 週）：PPT 解析、圖像渲染、ColPali 特徵提取、ImageBind 對齐
- **Lakehouse-Retrieval-Agent**（第 4-5 週）：LanceDB 索引構建、雙層檢索邏輯實現、全文檢索集成
- **Reasoning-Reranker-Agent**（第 5-6 週）：MM-R5 集成、思維鏈生成、評分函數設計
- **Argos-Verification-Agent**（第 6-7 週）：視覺定位、幻覺偵測、審計日誌、證據地圖生成

### Phase 3: Integration Testing（第 8-9 週）
1. 端到端集成測試：100 份投影片的查詢與驗證全流程
2. 性能基准測試：延遲分佈、吞吐量、資源消耗
3. 人工評估：召回率、精度、推理可解釋性、驗證準確度任務執行

### Phase 4: Canary Deployment to Production（第 10-12 週）
1. 灰度發佈到生產環境，初期流量 5% → 25% → 50% → 100%
2. 監控線上性能、用戶反饋、錯誤率，按需調整
3. 建立線上 A/B 測試框架，對比新舊系統的精度與用戶滿意度

### Rollback Strategy
- **關鍵時間點**：若 Canary 階段的 Precision@1 < 80% 或幻覺率 > 5%，立即回滾至上一版本
- **資料空間**：LanceDB 索引與原 OCR 索引並行保留，新舊系統可快速切換
- **消息隊列**：已發送但未處理的消息自動重定向至舊系統的消息隊列，確保無消息丟失

## Open Questions

1. **ImageBind 空間在簡報領域的對齐品質**：未知數值。建議先在小規模（~10K）標註投影片上驗證，評估 Consistency Score

2. **MM-R5 的貝葉斯推理能力**：MM-R5 對於涉及復雜統計或金融概念的投影片推理精度未知。需專家人工評估

3. **多向量索引的最優量化方案**：8-bit vs 16-bit，精度損失的具體數字需實驗驗証

4. **Argos 驗證層的計算成本與延遲**：單個驗證任務的計算複雜度在 500-1000ms 是否能穩定達到，需壓力測試

5. **跨語言投影片（中英混合）的聯合檢索**：需評估 ImageBind 在多語言統一空間中的精度

6. **與既有簡報管理系統（如 SharePoint / Confluence）的集成方案**：API 合同如何定義，認證與權限如何設計未來方向

建議留待第一個 Sprint 完成系統框架後，再深入攻關這些開放問題。
